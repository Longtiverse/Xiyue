为什么这样设计 AGENTS.md？

这个页面不仅展示 Xiyue 的 AGENTS.md 内容，更重要的是解释每一个栏目存在的原因——它是为了解决 AI 协作中的哪个具体问题而诞生的。

🎵 Xiyue 习乐 Monorepo Android + HTML Sandbox AI-First 设计
01

项目概述

设计目的
这是 AI 的「第一眼」。很多 AI 在拿到任务后，会先尝试理解项目是什么、技术栈是什么。如果没有这个栏目，AI 可能会把 Xiyue 当成一个普通的 Web App，而忽略掉它其实是一个"Android App + 共享 JS 核心"的 monorepo 结构。这里用一句话+ bullet points 明确主入口和共享逻辑的位置，让 AI 在动手写代码之前就建立正确的全局认知。
  • 主入口：apps/android（Kotlin + Jetpack Compose）
  • 共享逻辑：packages/music-core（纯 JavaScript，音乐理论核心）
  • 快速验证：apps/html-sandbox（HTML/JS 沙盒，用于内部测试与 UI 预览）
  • 包管理：npm Workspaces（packages/* + apps/*）
02

环境要求

设计目的
AI 的训练数据有截止日期，它可能不知道 Node.js 20 才原生支持 node --test，也可能默认推荐 Kotlin 1.9 的语法。这个栏目明确告诉 AI：「请以这些版本为基准生成代码」。它解决的核心问题是版本错位导致的兼容性问题——比如 AI 写出旧版 ESLint 配置、用废弃的 Kotlin API、或推荐 pnpm 导致 lock 文件冲突。
  • Node.js: ≥ 20（原生 node --test）
  • Kotlin: 2.0.21
  • JDK: 17
  • 包管理器: npm（禁止引入 pnpm/yarn 配置）
03

目录结构与模块规则

设计目的
这个栏目是为了解决 monorepo 中最隐蔽但最高频的错误：错误的模块引用方式和环境边界混淆。AI 很容易写出 ../../../packages/music-core/src/index.js 这样的深层相对路径，或者在纯 Node 包里引入 window 对象。通过把「目录树 + 模块规则」放在一起，并配上 ✅❌ 示例，AI 能更直观地理解边界在哪里。

模块引用规则

// ✅ 正确
import { generateScalePitches } from '#music-core';

// ❌ 错误
import { generateScalePitches } from '../../../packages/music-core/src/index.js';
  • packages/music-core 必须是纯 JS/Node，不能引入浏览器 API
  • apps/html-sandbox 可以使用浏览器 API

改动决策树

设计目的
这是整个 AGENTS.md 的「核心导航系统」。它的设计初衷是：AI 接收到的任务指令往往是局部的（比如"加一个节拍器"），但一个功能可能涉及音乐核心、Android UI、HTML 沙盒三个地方。决策树把「我该先改哪里、后改哪里、怎么验证」显性化，防止 AI 只改了一个端而漏掉其他端，或者改错顺序导致反复返工。

常用命令

设计目的
AI 在生成代码后，经常需要运行测试或 lint 来验证。但如果 AI 不知道项目有哪些 script，它要么去猜，要么硬编一个可能不存在的命令（比如 npm run test:unit）。这个栏目把所有高频命令集中列出，让 AI 在「写代码 → 验证」的闭环中不需要翻 package.json，直接复制使用。
npm test运行全部测试
npm run test:music-core仅测试核心库
npm run lintESLint 检查
npm run lint:fix自动修复 lint 问题
npm run formatPrettier 格式化
npm run typecheckTS 类型检查（music-core）
npm run build:android构建 Android APK
npm run sandbox启动 HTML 沙盒服务器

代码规范

设计目的
AI 的默认风格偏好可能与项目不一致（比如用双引号、4 空格缩进、或者 snake_case 命名）。这个栏目通过明确且简洁的规则 + 示例，让 AI 生成的第一版代码就更接近项目的实际风格，减少格式化返工。特别重要的是区分了 JS（2 空格）和 Kotlin（4 空格），这是很多 AI 容易混淆的地方。

JavaScript / Node.js

  • 模块：ESM（import/export）
  • 命名：文件 kebab-case.js，变量 camelCase，常量 UPPER_SNAKE_CASE
  • 风格：单引号、2 空格缩进、每行最多 100 字符

Kotlin（Android）

  • 4 空格缩进、类名 PascalCase
  • 每行最多 120 字符

提交规范

设计目的
当 AI 执行多步骤任务时（比如 Agent 模式），它可能会自动生成 git commit。如果没有提交规范约束，AI 的 commit message 往往很随意（比如 "update files"）。这个栏目让 AI 学会用 Conventional Commits，确保项目历史可读，也便于未来自动生成 CHANGELOG。 
feat(playback): add metronome support
fix(ui): correct root note selector alignment
test(music-core): add tests for pentatonic scales

常用类型：feat、fix、docs、style、refactor、perf、test、chore、ci

提交前检查清单

设计目的
这是从「规范」到「执行」的最后一道防线。AI 在完成任务时，容易因为用户催促或步骤限制而跳过测试和 lint。把检查项写成显式的 checklist，能让 AI 在 self-correction 阶段主动对照执行，也便于人类快速 review AI 的输出是否合规。这里特别强调 monorepo 特有的同步检查：改完 music-core 后，必须确认 Android 和 Sandbox 调用方是否适配。
npm test 全部通过
npm run lint 无错误
npm run format 已执行
若修改了 music-core 的公开 API，npm run typecheck 通过
新功能/修复包含对应测试
music-core API 变更已同步到 Android 和 Sandbox 调用方
没有手动修改 package-lock.json、builds/、.tmp/ 等生成目录

AI 协作注意事项

设计目的
这是 AGENTS.md 的「护城河」，专门收录那些"AI 特别特别容易踩的坑"。它的设计逻辑是：与其让 AI 犯错后再纠正，不如提前把陷阱标出来。这里每一条都对应一个真实的高频错误场景。把它们集中放在文档末尾，是因为前面的栏目负责"教 AI 怎么做"，而这个栏目负责"告诉 AI 什么不能做"。
🚫 禁止手改的文件
package-lock.json 由 npm install 生成；builds/、.tmp/、coverage/ 是生成目录；Gradle wrapper 和设计规范文档也不要手改。
⚠️ test:android 是 JS 测试，不是 Kotlin 单元测试
它运行在 Node.js 下，测试路径是 apps/android/test/**/*.test.js。真正的 Android 编译用 npm run build:android 或 Android Studio。
⚠️ packages/music-core 不能依赖浏览器 API
因为它是纯 Node/JS 库，被 Android 和 Sandbox 两端引用。引入 window、document、navigator 会导致测试或运行时失败。
⚠️ Android 状态管理与 music-core 数据结构需兼容
Android 使用 Action/Reducer/State 状态管理。如果 music-core 返回的数据结构变了，Kotlin 侧的解析逻辑和状态定义必须同步更新。
⚠️ 不要引入新的包管理器配置
项目只用 npm。不要创建 pnpm-lock.yaml、yarn.lock、.yarnrc 等文件。
⚠️ 环境变量从 .env 读取，不要硬编码本地路径
Android 构建依赖 .env 中的 JAVA_HOME 和 ANDROID_SDK_ROOT。脚本里不要写死 C:\Users\... 这类绝对路径。
