1
📡 会话启动与上下文恢复
Plan
用户说"继续"时，不要从零猜测。按协议读取进度文件、计划文件和 Git 历史。
标准恢复协议
  • 读取 .tmp/session-progress.md
  • 扫描 .tmp/plan-*.md 找出未完成的计划
  • 查看最近 3-5 条 git commit
  • 向用户汇报状态并确认下一步
黄金法则：如果进度文件不存在，就基于计划文件；如果连计划文件也没有，就基于 git log；如果三者都没有，明确告知用户"没有找到可接续的上下文"。

2
🏗️ 架构与影响面分析
Arch
写代码前先回答三个问题：状态存在哪？配置变更后会丢失吗？生命周期边界是什么？
架构先于实现检查清单
状态保存在哪里？（remember / rememberSaveable / ViewModel / Service）
屏幕旋转后状态会丢失吗？
生命周期边界是否清晰？（Composable / Service / 进程被杀）
如果现有架构无法承载，是否先提出调整方案？
禁止：在状态管理方案未确定前，先写「看起来完整」的 UI。
↓
3
📝 制定计划（Plan-First）
Plan
超过 3 个步骤、涉及多模块、或预计跨会话的任务，必须写 .tmp/plan-{task}.md。
必须写计划的情况（满足任意一条）
  • 预计超过 3 个步骤
  • 涉及 2 个及以上模块/目录
  • 预计耗时超过 1 个完整会话
  • 存在需要明确的 验收标准
计划文件必备字段
  • Goal：一句话描述目标
  • Acceptance Criteria：验收标准 checklist
  • Steps：按顺序列出，标注涉及文件
  • Dependencies：前置依赖或阻塞点
  • Risks：风险点及规避策略
↓
4
🛠️ 科学编码执行
Exec
遵循效率原则：最小改动、Sandbox 优先、批量处理、独立任务并行化。
效率原则
  • 最小改动：只改与任务直接相关的文件
  • Sandbox 优先：UI/动画先在 apps/html-sandbox 验证
  • 精准测试：优先跑相关子测试，最后才全量
  • 批量处理：同主题文件一次性修改，减少往返
  • 避免过度工程：简单方案优先，需要时再抽象
并行化决策
YES相互独立的搜索、验证、文件生成 → 使用子代理并行
NO有数据依赖、修改同一文件、需先统一接口 → 串行执行
↓
5
🛡️ 敏感子系统敬畏清单
Check
涉及音频、Service、Native 资源时，主路径编译通过只是开始，边界情况才是第一优先级。
音频 / Service / JNI 强制检查项
资源是否正确释放？（release() / close() / unregister()）
是否存在竞态条件？（start/stop/cancel 时序、快速连点）
异常路径是否有降级方案？（try-catch + fallback）
配置变更后是否会导致重复初始化或资源泄漏？
禁止：仅在「主路径编译通过」时就认为敏感子系统任务完成。
↓
6
✅ 真实场景验证
Check
编译通过 ≠ 可用。必须在模拟器/真机上验证生命周期和边界场景。
真实场景验证清单（至少选 3 项）
屏幕旋转 / 配置变更（状态是否丢失？音频是否中断？）
切到后台再返回（Service 是否还在？UI 是否同步？）
进程被系统杀死后恢复（数据是否蒸发？）
低内存 / 音频焦点被抢占（是否有优雅降级？）
快速重复操作（按钮连点、Tab 快速切换）
冷启动路径（首次安装用户是否能正确进入？）
验证结果必须记录在 session-progress.md 或 plan 文件中。如因环境限制无法验证，必须明确说明并评估风险。
↓
7
💾 状态持久化与 Git 断点
Persist
每完成一个关键步骤，或每次会话结束前，必须更新进度文件并提交代码。
必须做的两件事
  • 更新 .tmp/session-progress.md：Completed / In Progress / Next Steps / Modified Files
  • 用 wip(scope): ... 或 checkpoint(scope): ... 提交当前代码
状态外化原则：所有超过当前会话可能保留时间的关键信息，都必须写入文件。代码注释、plan 文件、progress 文件、Git commit，是 AI 的四个外置硬盘。
禁止在 WIP 提交中使用 feat: 或 fix:，避免混淆正式版本历史。
↓
8
📦 任务完成与归档
Archive
所有验收标准通过、代码合并、相关文档更新后，将计划文件归档到 .tmp/archive/。
归档流程
  • 确认 plan 中所有 Acceptance Criteria 和 Steps 已勾选
  • 运行最终检查：npm test、npm run lint、npm run format
  • 将 plan 文件移动到 .tmp/archive/done-{task}-{YYYY-MM-DD}.md
  • 更新 session-progress.md 标记当前任务完成，并写下下一个任务的启动信息
归档不是删除历史，而是为下一个任务腾出清晰的桌面。归档后的 plan 文件仍可作为项目 retrospectives 和最佳实践沉淀的素材。

这套流程的核心只有一句话：

"把 AI 的大脑从对话窗口迁移到文件系统 + Git 历史 + 标准协议上。"
