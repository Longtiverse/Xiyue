│   - Quick-start/current practice card.                                       │

│   - Collapsed or softened library access.                                    │
│   - Minimal, calm control density.                                           │

│   - Practice card swap/selection emphasis.                                   │
│   - Library item selection transition.                                       │
│   - Inline note about reduced-noise selection behavior.                      │

│   - Expanded current-practice hero / now-playing view.                       │

│   - Current note emphasis.                                                   │
│   - Keyboard preview with active key animation.                              │
│   - Waveform / oscilloscope panel running.                                   │
│   - Muted gold live-state accents.                                           │
│ 4. Detailed playback state                                                   │
│   - Show progression from minimal note-focus to richer detail.               │
│   - Sequence chips / progress / loop / BPM status.                           │
│   - Subtle explanatory callout about tap-to-expand detail.                   │

│   - Show how full browsing appears without overwhelming the home screen.     │
│   - Recent, filters, list treatment, selected item state.                    │
│   - Animated transition or slide-up panel treatment.                         │
│                                                                              │
│ These can be represented as stacked phone mock frames on one page or as a    │
│ single main phone frame whose contents auto-transition while smaller preview │
│  frames or progress markers show the journey.                                │

│ Prototype implementation strategy                                            │
│                                                                              │
│ 1. Restructure the page around phone presentation                            │
│                                                                              │
│ Turn the current wide desktop composition into a mobile-centered review      │
│ page:                                                                        │
│ - Put one primary phone frame at the center of the page.                     │
│ - Optionally add a vertical state rail, progress dots, or compact thumbnails │
│  beside/below it.                                                            │
│ - Keep the outer page subdued so attention stays on the simulated handset.   │
│ - Use inline callouts attached to the frame edges to explain what changed in │
│  each state without feeling like documentation-first output.                 │
│                                                                              │
│ 2. Encode the interaction storyboard                                         │
│                                                                              │
│ Create a lightweight state machine in plain JS that cycles through named     │
│ stages such as:                                                              │
│ - ready                                                                      │
│ - select-root                                                                │
│ - select-practice                                                            │
│ - playback-focus                                                             │
│ - playback-detail                                                            │
│ - browse-library                                                             │
│                                                                              │
│ For each stage, toggle data attributes/classes on the page to drive:         │
│ - visible screen sections                                                    │
│ - active selections                                                          │
│ - button/live states                                                         │
│ - callout copy                                                               │
│ - motion cues                                                                │
│ - progress indicators                                                        │
│                                                                              │
│ The loop should auto-play, pause briefly on each state, and optionally       │
│ reset/replay.                                                                │
│                                                                              │
│ 3. Make the phone UI feel final, not schematic                               │
│                                                                              │
│ Refine the mockup into an actual phone presentation:                         │
│ - realistic handset frame / safe-area spacing                                │
│ - bottom reachability and thumb-friendly controls                            │
│ - section density closer to Android phone use, not desktop cards shrunk down │
│ - one clear primary action per state                                         │
│ - library and secondary controls revealed progressively instead of displayed │
│  all at once                                                                 │
│                                                                              │
│ 4. Simulate the key interactions the user asked for                          │
│                                                                              │
│ Selection transitions                                                        │
│                                                                              │
│ - Root chips slide or shift emphasis when the selected note changes.         │
│ - Practice card changes with fade/slide transition.                          │
│ - Library reveal uses a soft bottom-sheet or expanding panel motion.         │
│ - Selected items get jade edge/border/fill emphasis, not loud glow.          │
│                                                                              │
│ Keyboard animation                                                           │
│                                                                              │
│ - Animate active keys using scale, brightness, and gold live fill inspired   │
│ by AnimatedKeyboardKey.kt.                                                   │
│ - Sequence through a few notes during playback states.                       │
│ - Keep inactive keys matte and subdued.                                      │
│ - Consider a ghost/preview highlight for upcoming notes.                     │
│                                                                              │
│ Waveform / oscilloscope effect                                               │
│                                                                              │
│ - Add a self-contained waveform using CSS bars, SVG path animation, or       │
│ Canvas.                                                                      │
│ - Tie its apparent motion speed to the displayed BPM.                        │
│ - Calm idle state: near-flat or low-amplitude breathing line.                │
│ - Playback state: richer oscillation plus slight fill/glow.                  │
│                                                                              │
│ Live playback aura                                                           │
│                                                                              │
│ - Add restrained edge glow or card aura inspired by MetronomeEdgeGlow.       │
│ - Use muted gold only during active playback.                                │
│ - Keep animation soft and rhythmic rather than flashy.                       │
│                                                                              │
│ Display mode transition                                                      │
│                                                                              │
│ - Show the playback card shifting from minimal note-focus to detailed mode,  │
│ inspired by PlaybackDisplaySection.kt.                                       │
│ - Use one simple tap-to-expand explanatory callout.                          │
│                                                                              │
│ 5. Explain each page/state without cluttering the design                     │
│                                                                              │
│ Use inline callouts that are visually secondary:                             │
│ - small anchored labels outside or beside the phone frame                    │
│ - short design notes like “Primary CTA stays singular here” or “Selection    │
│ stays local; no full-page context switch”                                    │
│ - one focused message per state                                              │
│ - avoid long paragraphs inside the phone UI itself                           │
│                                                                              │
│ A compact summary strip below the phone can still list:                      │
│ - state name                                                                 │
│ - intent                                                                     │
│ - motion behavior                                                            │
│ - related Android component inspiration                                      │
│                                                                              │
│ 6. Keep it self-contained                                                    │
│                                                                              │
│ - No external fonts, JS frameworks, animation libs, or hosted assets.        │
│ - Use local HTML/CSS/JS only.                                                │
│ - Ensure direct file:/// opening still works.                                │
│ - Prefer CSS keyframes + tiny JS orchestration over heavy scripting.         │
│                                                                              │
│ Critical implementation details                                              │
│                                                                              │
│ 1. This remains a review artifact, not app production code.                  │
│ 2. The deliverable should communicate final mobile feel, not just a mood     │
│ board.                                                                       │
│ 3. The visual system should continue the quieter premium direction:          │
│   - charcoal/slate base                                                      │
│   - jade/mint primary emphasis                                               │
│   - muted gold for live playback only                                        │
│ 4. The default visible state should feel minimal and fast to start from.     │
│ 5. Interaction staging should clearly demonstrate:                           │
│   - selection changes                                                        │
│   - play/pause emphasis                                                      │
│   - keyboard note animation                                                  │
│   - waveform/oscilloscope motion                                             │
│   - playback detail expansion                                                │
│   - library reveal behavior                                                  │
│ 6. The page should still degrade gracefully at narrower widths and remain    │
│ viewable in a desktop browser.                                               │
│ 7. If JS is added, it should be tiny, declarative, and easy to inspect.      │
│                                                                              │
