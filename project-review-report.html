架构亮点: HomeReducer (MVI 模式)

HomeReducer.kt Kotlin 
class HomeReducer(private val stateFactory: HomeStateFactory = HomeStateFactory()) {
    fun reduce(state: HomeUiState, action: HomeAction): HomeUiState {
        // ...辅助函数 nextState(...)
        return when (action) {
            is HomeAction.SelectLibraryItem -> nextState(
                selectedLibraryItemId = action.itemId,
                isLibraryOverlayVisible = false,
            )
            is HomeAction.TogglePlayback -> nextState(
                isPlaying = !state.isPlaying,
                isPaused = state.isPlaying,
            )
            is HomeAction.SyncPlaybackSnapshot -> nextState(
                playbackSnapshot = action.snapshot,
            )
            // ... 其他 Action 处理
        }
    }
}

采用纯函数 Reducer 处理用户意图(Action)，通过 StateFactory 重新构建不可变状态，符合 MVI 单向数据流原则。

⚠ 构建配置观察

  • Debug 构建启用 ProGuard：debug 类型设置了 isMinifyEnabled = true 和 isShrinkResources = true，这会延长构建时间并影响调试堆栈跟踪的可读性。
  • isDebuggable = false：debug build 关闭了可调试性，实际上更像是 release 配置。建议在开发调试时恢复 isDebuggable = true 并关闭 minify。
  • APK 命名规范：使用 SimpleDateFormat("yyyyMMdd-HHmm") 自动生成带时间戳的 APK 文件名，便于版本追溯。

! 发现的问题

1. Android Kotlin 缺乏真正的单元测试

高优先级

现有 Android 测试使用 Node.js 脚本验证文件存在性和结构，无法验证 Reducer 逻辑、音频合成正确性或 Compose UI 行为。建议引入 JUnit + Robolectric/Compose Test。

2. Debug 构建配置异常

中优先级

build.gradle.kts 中 debug buildType 启用了 isMinifyEnabled=true 和 isDebuggable=false，几乎等同于 release 配置。这会显著拖慢日常开发构建并阻碍调试。

3. library.json 数据重复

中优先级

音阶/和弦数据同时存在于 packages/music-core/data/library.json 和 apps/android/app/src/main/assets/library.json。虽然内容可能相同，但存在同步不一致的风险。建议通过构建脚本自动复制。

4. 单一贡献者风险

低优先级

所有 19 次提交均来自 Longtiverse。知识集中度高，建议完善 CONTRIBUTING.md 流程并考虑代码审查机制（PR + Review）。

5. 部分 JS 覆盖率偏低

低优先级

storage.js (66.4%)、open-html-sandbox.js (35.5%)、controller.js (70.3%) 覆盖率较低。多为错误处理路径和浏览器环境特定代码未触发。

✓ 改进建议

🎯 短期 (1-2 周)

  • 修复 debug buildType 配置：关闭 minify，开启 isDebuggable
  • 在 CI 中添加 Android instrumentation test 步骤
  • 添加 Gradle 任务自动同步 library.json 到 Android assets

🚀 中期 (1-2 月)

  • 建立真机/模拟器 UI 自动化测试（Compose Test + Maestro）
  • 引入 SonarQube 或 Detekt 做 Kotlin 静态分析
  • 完善错误上报和崩溃监控（Firebase Crashlytics）

🔮 长期 (3-6 月)
